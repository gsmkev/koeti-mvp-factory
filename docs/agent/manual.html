Onboarding · AFK SaaS software factory

Cómo funciona el factory, de punta a punta

Una fábrica de MVPs SaaS: infraestructura compartida en @koeti/*, lógica de negocio por app, y un loop autónomo que va de un párrafo a un PR verificado. Este manual reúne los invariantes del sistema y cómo se opera.

Parte I · Arquitectura — qué es el sistema
01

Topología de despliegue

Dónde corre cada cosa y con qué servicios externos habla. El dev trabaja contra una VM en OCI (donde vive este agente y los dev servers), alcanzada desde el navegador por Tailscale + port-forward de VS Code. Producción es Vercel detrás de Cloudflare.

02

Grafo de dependencias — packages ↔ apps

La regla de oro del monorepo: las apps importan solo de @koeti/*; nunca de otra app.

db es la raíz; auth y billing dependen de ella. ui, email, analytics y ai son hojas. config es dev-only (tsconfig/eslint). @koeti/ai va marcado planned: aún no existe en packages/.

03

Modelo de datos — ERD

El schema base (packages/db/src/schema.ts). Cada MVP añade sus propias tablas de negocio con una columna team_id, y así crudActions las scope-a por equipo automáticamente.

04

Auth + sesión

Password u OAuth de Google desembocan en una cookie JWT httpOnly de 1 día. Cada request posterior re-verifica el token y scope-a la query por equipo.

05

Capa de IA — OCR de recibo planned

Primer consumidor previsto: gastos. Patrón no-op-sin-key idéntico a @koeti/email y @koeti/analytics: sin API key de Mistral, ocr() devuelve null y el MVP cae a carga manual.

06

Ciclo del factory

De una idea en un párrafo a un MVP con PR. El detalle de las 7 fases está en la sección 13.

07

Billing — ciclo de vida Stripe

Checkout y portal del cliente en Stripe; el webhook sincroniza el estado de suscripción de vuelta a la tabla teams. El gating de plan es isSubscribed(team) = active || trialing.

Gap conocido: el webhook solo maneja subscription.updated/deleted. Falta invoice.payment_failed y checkout.session.completed — añadir cuando importe el dunning.

08

Seguridad — RBAC + multi-tenancy

Dos guardas se combinan en cada mutación: quién sos (rol) y de qué equipo (tenant). Ninguna query de negocio corre sin ambos.

  • RBAC — requireRole('viewer') en páginas, withTeam(fn,'admin') en actions. crudActions recibe minRole (default member).
  • Tenancy — crudActions inyecta team_id en todo insert y lo filtra en update/delete: olvidar el scope es imposible. Las lecturas del MVP deben repetir el WHERE team_id a mano.
  • Superadmin — isSuperadmin(user) vía env SUPERADMIN_EMAIL, fuera de la jerarquía de equipo.
09

CRUD team-scoped

El crudActions cubre el ~80% de cada MVP: valida con Zod, exige rol vía withTeam, inyecta team_id y revalida. Entidades con lógica extra llevan una action escrita a mano al lado — no forzar todo por acá.

10

Integración MVP ↔ MVP

Dos únicas vías permitidas, nunca import entre apps: datos por HTTP (route handlers + API key de equipo, Authorization: Bearer koeti_…) y navegación por deep-links con estado en la URL (nuqs).

12

Reglas que se auto-aplican por path

Cada regla en .claude/rules/* declara paths: (globs). Cuando el agente toca un archivo que matchea, la regla se inyecta en contexto sin pedirla — el patrón correcto llega solo al editar la zona correcta.

13

Workflow autónomo — el loop /factory

Siete fases, cero preguntas. La Fase 5 (Verify) es el gate: es la Definition of Done y nada sale sin que pase. /create-saas es la variante que arranca en Implement cuando ya hay spec+plan.

14

Mapa de comandos

El día a día: setup → crear/iterar → gate.

Regla de cierre: antes de decir que algo funciona, typecheck && test && build en verde. Tocaste pages/actions/schema → sumá verify-app + e2e-app. Tocaste saas-template o create-mvp.mjs → sumá smoke.

Qué no está en este manual — a propósito

Se lee directo del código o de las reglas; documentarlo sería deuda. Estructura de carpetas y estándares de imports viven en CLAUDE.md; los patrones de charts / URL-state / CRUD en .claude/rules/*.md; la API exacta de cada package en sus index.ts. Acá va solo lo transversal: topología, dependencias e invariantes de seguridad.
