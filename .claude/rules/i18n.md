# i18n — every app speaks en / es / pt

The factory ships trilingual. Locale is chosen by a **cookie** (`NEXT_LOCALE`),
no `/es` URL prefix, so `proxy.ts` and every deep link stay untouched. Powered
by [`next-intl`](https://next-intl.dev) in its "without i18n routing" mode.

## The split

- **`@koeti/i18n`** owns the plumbing and the **shared baseline** messages
  (`messages/{en,es,pt}.json`): auth, dashboard nav, settings pages, billing,
  action errors, error pages. Every app inherits these for free.
- **Each app** ships only its **business** messages in `app/../messages/{en,es,pt}.json`
  (namespaces for its own features). `i18n/request.ts` merges baseline + app
  per request — app keys win on collision.

New apps get all of this by `pnpm create-mvp` copying the template. Nothing to wire.

## Adding a string

1. Add the key to **all three** files of the right layer (baseline in
   `@koeti/i18n`, business in the app). The key-parity test fails CI if you miss one.
2. Read it:
   ```tsx
   // server component / server action / route handler
   import { getTranslations } from 'next-intl/server'
   const t = await getTranslations('nav')
   t('overview')

   // client component
   import { useTranslations } from 'next-intl'
   const t = useTranslations('nav')
   t('overview')
   ```
3. Interpolation `t('currentPlan', { plan })`, plurals via ICU in the JSON
   (`"{count, plural, one {# item} other {# items}}"`), arrays via `t.raw('key')`.

## Rules

- **Cookie-only, no schema.** Locale lives in `NEXT_LOCALE` (1-year cookie set
  by `<LocaleSwitcher/>`). No `users.locale` column — add one only if you need
  cross-device persistence.
- **Every user-facing string goes through `t()`.** Brand names, code-like tags
  (`[auth]`), and Stripe-provided plan names stay literal.
- **`<LocaleSwitcher/>`** (from `@koeti/i18n`) belongs in the dashboard sidebar
  footer and the marketing/login header. It's a native `<select>` that sets the
  cookie and `router.refresh()`es.
- **Pages become dynamic** (`ƒ`) because rendering depends on the cookie — that's
  inherent to cookie i18n, not a bug.
- **`<html lang>`** must be the resolved locale: `lang={await getLocale()}` in the
  root layout, which also wraps children in `<NextIntlClientProvider>`.

## Emails

react-email renders outside the request, so there's no next-intl context.
Email copy lives as a small per-locale `copy` dict inside each template in
`@koeti/email`; each template takes a `locale` prop and exports a `*Subject(locale)`
helper. Callers pass the acting user's locale: `const locale = await getLocale()`.

## Testing

Each layer has a **key-parity test** asserting `en`/`es`/`pt` carry the exact
same key set — a forgotten translation fails there instead of rendering the raw
key in production. Keep it; it's the cheapest guard against drift.
